# PRD: EA Network & Referral System (Job Creation Initiative)

## 1. Overview
"Tod" (an AI recruiter) will source Executive Assistants (EAs) via LinkedIn to join Buy Anything as part-time distributed EAs. EAs will onboard via a simple SMS login, receive a unique "copylink," and distribute it to their networks. When a new user signs up via an EA's link, the EA will receive a **30% revenue share** on all revenue generated by that user.

**Mission:** Job creation in the age of AI. Providing a commission-based revenue stream for displaced white-collar workers (targeting ~$25/hr equivalent).

## 2. Core User Flows

### A. EA Onboarding
1. EA receives cold outreach from "Tod".
2. EA visits a dedicated landing page (e.g., `buy-anything.com/ea`).
3. EA logs in via Cell + SMS code (leveraging our existing auth).
4. EA completes Stripe Connect onboarding to enable payouts.
5. EA is granted "EA Status" and provided a unique referral link (`buy-anything.com/?ref=ea_12345`).

### B. The Copylink & Attribution (The Hard Part)
1. User clicks the EA's referral link.
2. The referral code is captured and persisted (cookie/local storage).
3. When the user eventually authenticates/registers (via Cell + SMS), the referral code is locked into the database as a permanent attribution.
4. If the user creates an account, they are forever tied to the referring EA.

### C. Revenue Share & Payouts
1. The referred user completes a transaction where Buy Anything takes a platform fee.
2. The backend intercepts the checkout/payment event.
3. The system splits the platform fee: 70% to Buy Anything, 30% transferred to the referring EA's Stripe Connect account.
4. EAs can view their referral metrics and earnings balance on an EA Dashboard.

## 3. Deep Dive: Referral Tracking Strategy

Referral tracking is historically brittle due to ad-blockers, Safari's ITP, and cross-device drops. Here is how we ensure EAs get paid accurately:

### The Tracking Mechanism
1. **URL Parameter Capture:** When a request hits `/?ref=<CODE>`, the frontend proxy/middleware intercepts it immediately.
2. **First-Party Cookie (HttpOnly):** We set a first-party cookie (`ba_ref_code`) with an expiration of 90 days. Because it is first-party and server-set, Safari ITP and most ad-blockers will not strip it.
3. **Local Storage Fallback:** We also push the code to `localStorage` as a redundant backup.
4. **Registration Hook:** During the Twilio SMS verification flow, the frontend passes the referral code along with the `POST /auth/verify` request.
5. **Database Lock-in:** The backend reads the referral code, validates it belongs to an active EA, and sets `User.referred_by_id = ea_user_id`. This is a one-time, immutable write. 

### Edge Cases & Mitigations
*   **Cross-device drop-off:** (e.g., User clicks on phone, but signs up later on laptop). *Mitigation:* We cannot easily fix this without device fingerprinting. EAs should be educated to encourage immediate login/sign-up. 
*   **Self-Referral Fraud:** EAs using their own link to get 30% off their own purchases. *Mitigation:* Prevent `referred_by_id` from matching the user's own ID, and flag identical payment methods or IPs.
*   **Last-Touch vs First-Touch:** If a user clicks EA 1's link, then EA 2's link, who gets the credit? *Decision:* **Last-touch attribution**. The most recently clicked link prior to account creation wins.

## 4. Technical Architecture

### Database Schema Updates (Alembic Migration required)
*   **`User` Table Updates:**
    *   `is_ea: bool` (default `False`)
    *   `ea_referral_code: str` (unique, nullable, generated when `is_ea=True`)
    *   `referred_by_id: uuid` (FK to `User.id`, nullable)
    *   `stripe_connect_account_id: str` (nullable)

### Stripe Integration
*   We will use **Stripe Connect (Express or Custom accounts)**.
*   When charging the referred user, we will use **Separate Charges and Transfers** (or Destination Charges) to automatically route 30% of the platform fee to the `stripe_connect_account_id` of the user's `referred_by_id`.

### Required API Endpoints
1.  **`POST /ea/onboard`**: Upgrades a user to an EA, generates their referral code, and returns a Stripe Connect onboarding link.
2.  **`GET /ea/dashboard`**: Returns metrics for the EA (number of signups, total volume, total earned).
3.  **`POST /auth/start` & `/auth/verify`**: Must be updated to accept an optional `ref_code` parameter.

## 5. Phase 1 Implementation Steps
1.  Add DB columns for tracking (`referred_by_id`, `ea_referral_code`, `is_ea`).
2.  Implement the first-party cookie + localStorage referral tracking on the Frontend.
3.  Update the Auth flow to consume and save the referral code on account creation.
4.  Build the EA Dashboard & Copylink generation UI.
5.  Integrate Stripe Connect API for EA onboarding and automated 30% transfers.
