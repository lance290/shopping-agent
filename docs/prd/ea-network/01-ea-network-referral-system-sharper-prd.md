# PRD: EA Network & Referral System

**Audience:** Technical founder / engineering lead  
**Status:** Draft for Phase 1  
**Product:** Buy Anything  
**Primary goal:** Acquire new customers through a distributed network of Executive Assistants (EAs) and pay them commissions reliably with minimal manual ops.

---

## 1. Summary

Buy Anything will recruit Executive Assistants (EAs) into a lightweight referral partner program.

Each EA will:
- sign up with existing SMS auth
- complete Stripe Connect onboarding
- receive a unique referral link and referral code
- invite people from their network
- earn commission on qualified platform-fee revenue generated by referred users

The system must solve three things well:
1. **Attribution:** correctly tie a new user to an EA
2. **Commission accounting:** calculate pending, vested, paid, reversed, and disputed earnings
3. **Payouts:** send funds to EAs through Stripe Connect without manual bookkeeping

---

## 2. Product Goals

### Goals
- Launch an EA referral engine that can be implemented quickly using existing auth and payments infrastructure
- Create a simple EA experience: onboard, get a link, track signups, get paid
- Support durable attribution with a low-support operational model
- Maintain a clean ledger of commissions for finance and support

### Non-goals for Phase 1
- Multi-touch attribution modeling
- Team hierarchies or manager overrides
- Promo discounts tied to referral codes
- Advanced tax reporting UX inside our app
- International expansion beyond Stripe-supported/approved operating regions

## 3. Core Business Rules

### 3.1 Roles
- **EA:** approved referrer with an active Stripe Connect account
- **Referred user:** user whose first completed account registration is attributed to an EA
- **Admin:** internal operator with ability to review fraud flags, reverse commissions, and resolve disputes

### 3.2 Attribution rule
- Attribution is set when a new user completes account creation / phone verification.
- Attribution is stored as `referred_by_user_id` on the user record.
- Attribution is immutable after set, except by internal admin fraud/dispute tooling.
- Attribution model for Phase 1 is **last-touch before account creation**.

### 3.3 Commission rule
An EA earns commission only on **qualified platform fee revenue**.

**Definition:**
> EA commission = **30% of Buy Anything net platform fee revenue** collected on completed, non-refunded, non-disputed transactions attributable to that referred user.

### 3.4 Exclusions from commission base
The following are **not** commissionable unless explicitly changed later:
- product GMV
- sales tax
- shipping reimbursements
- pass-through payment processor fees
- tips
- fully refunded transactions
- fraudulent or disputed transactions

### 3.5 Commission state machine
Each commission record moves through these states:
- `pending` — payment succeeded, but still inside hold/refund window
- `vested` — hold window expired and eligible for payout
- `paid` — transfer created to connected account
- `reversed` — removed because of refund, dispute, fraud, or admin adjustment
- `failed` — payout attempt failed and needs remediation

### 3.6 Recommended payout timing
For Phase 1, **do not transfer commission instantly at checkout**.

Instead:
1. charge customer on the platform account
2. calculate commission and record a `pending` ledger entry
3. wait through a configurable hold window (recommended: **14 days** for normal card payments)
4. create the Stripe transfer only when commission vests

This is operationally safer than immediately transferring funds and then trying to unwind them after refunds or disputes.

## 4. User Flows

### 4.1 EA onboarding flow
1. EA receives outreach from Tod / recruiting workflow.
2. EA visits `buy-anything.com/ea`.
3. EA authenticates with existing SMS flow.
4. EA accepts referral-program terms.
5. System creates or links a Stripe Connect connected account.
6. EA completes Stripe-hosted onboarding.
7. System marks EA as `is_ea = true` once onboarding is complete enough to receive transfers.
8. System generates:
   - unique referral link: `https://buy-anything.com/?ref=ea_abc123`
   - short referral code: `ABC123`
9. EA lands on dashboard with copy/share CTA.

### 4.2 Referral acquisition flow
1. Prospect clicks EA referral link.
2. Middleware captures `ref` query parameter.
3. System stores referral signal in:
   - first-party cookie
   - local storage backup
   - optional session state
4. If prospect creates an account, frontend sends `ref_code` on auth verify / registration completion.
5. Backend validates code and writes `referred_by_user_id`.
6. If no cookie survives, user may manually enter the short referral code during signup.

### 4.3 Purchase and commission flow
1. Referred user completes a qualifying transaction.
2. Platform payment succeeds.
3. Backend computes platform fee and EA commission amount.
4. Commission ledger entry is created in `pending` state.
5. Refund/dispute window elapses.
6. Background job marks entry `vested`.
7. Payout job creates Stripe transfer to connected account.
8. Ledger entry becomes `paid` once transfer is confirmed.

## 5. Referral Attribution Design

### 5.1 Tracking strategy
Capture the referral as early as possible and persist it redundantly.

**Mechanism:**
1. Read `?ref=<CODE>` at edge/middleware level
2. Validate format before storage
3. Set first-party cookie `ba_ref_code`
4. Mirror to local storage for browser fallback
5. Pass `ref_code` into auth completion request
6. Persist immutable attribution on user creation

### 5.2 Attribution precedence
When account creation occurs, choose attribution source in this order:
1. explicit `ref_code` field submitted during signup
2. valid server cookie
3. local-storage-derived value submitted by frontend
4. none

### 5.3 Edge cases
#### Cross-device signup
If user clicks on phone and signs up later on laptop, cookie attribution may be lost.

**Phase 1 mitigation:**
- give each EA a short manual code
- allow code entry on signup
- expose support/admin override for verified cases

#### Multiple referrals
If user clicks multiple EA links before signup, most recent valid code before account creation wins.

#### Self-referral
Do not allow commission where:
- referred user ID equals referring EA user ID
- same verified phone number
- same payout account / same tax identity
- high-confidence duplicate-payment-method match

#### Existing users
Referral attribution only applies to **new account creation**. Existing users cannot later become attributed through a referral link.

## 6. Stripe Setup

### 6.1 Recommended Stripe model
Use:
- **Stripe Connect Express accounts** for EAs
- **separate charges and transfers** for the funds flow

### Why this is the right fit
- EAs are lightweight partners, not full merchants
- Stripe-hosted onboarding reduces compliance and verification UI work
- Platform remains the merchant of record for customer charges in the default flow
- Platform can pay one or more connected accounts from platform balance when commissions vest

### 6.2 Account type decision
For Phase 1, choose **Express**, not Custom.

Use Express because:
- Stripe handles onboarding, identity verification, and payout account collection
- EAs get limited Stripe-hosted account access
- engineering effort is materially lower
- it preserves more control than Standard while avoiding Custom-account overhead

Custom should only be considered later if you need a fully bespoke onboarding and payout-management surface.

### 6.3 Stripe object model
Per EA, store:
- `stripe_connect_account_id`
- `stripe_connect_onboarding_status`
- `stripe_charges_enabled`
- `stripe_payouts_enabled`
- `stripe_requirements_due_json`
- `stripe_default_currency`
- `stripe_country`

Per commission payout, store:
- `stripe_transfer_id`
- `stripe_transfer_reversal_id` (nullable)
- `stripe_payout_id` (nullable, if tracked downstream)

### 6.4 Connected account creation flow
When a user opts into the EA program:
1. backend creates Stripe connected account if one does not exist
2. request the minimum capabilities needed for payouts/transfers
3. create an Account Link for Stripe-hosted onboarding
4. redirect user to Stripe onboarding
5. on return, fetch account state and update internal status
6. do not mark EA payout-ready until required capabilities are active / requirements satisfied

### 6.5 Account onboarding endpoints
Recommended server endpoints:
- `POST /ea/onboard/start`
  - create or fetch connected account
  - create account link
  - return redirect URL
- `GET /ea/onboard/return`
  - fetch Stripe account
  - sync status to DB
  - redirect to EA dashboard
- `POST /ea/onboard/refresh`
  - generate a fresh account link if prior link expired

### 6.6 Payment and commission design
#### Recommended flow
1. Customer pays the platform via existing Stripe checkout/payment flow.
2. Platform records:
   - order ID
   - payment intent / charge ID
   - platform fee amount
   - referred EA ID if present
3. Commission engine creates a `pending` commission row.
4. After vesting, payout job creates a `transfer` from platform to connected account.

#### Why not transfer instantly?
Because refunds and disputes are expensive to unwind operationally. If you transfer immediately, you may need to reverse transfers later and handle failures or insufficient balances.

### 6.7 Payout scheduling
Recommended Phase 1 behavior:
- commission vests after hold window
- payout job runs daily
- group all vested commissions per EA into a transfer batch or transfer individually with metadata
- show payout totals in dashboard even if bank arrival is still pending inside Stripe

### 6.8 Webhooks to subscribe to
At minimum, handle:
- payment success event(s) used by your existing checkout path
- refund event(s)
- dispute/chargeback event(s)
- `account.updated`
- `payout.paid`
- `payout.failed`
- optionally `transfer.created` / `transfer.reversed` if you want stronger reconciliation

### 6.9 Stripe failure handling
If onboarding is incomplete:
- EA can see dashboard but cannot receive payouts
- commissions can accrue as `pending` or `vested_unpaid`
- payouts are blocked until account becomes eligible

If payout fails:
- mark payout batch / commission as `failed`
- notify EA to re-complete Stripe account details
- allow retry after account update

### 6.10 Tax / compliance note
Stripe can help collect payout identity data, but tax reporting, program terms, and classification of EAs as referral partners or contractors still need finance/legal review.

## 13. Open Decisions

These should be decided before engineering begins:
1. Is the hold window 7, 14, or 30 days?
2. Are commissions paid per transaction, daily batch, or weekly batch?
3. Are EAs approved automatically after onboarding, or manually reviewed?
4. Are existing users ever eligible for referral reassignment? Recommendation: **no**.
5. Do we allow referrals from all channels, or only approved outreach channels?
6. What are the exact program terms for fraud, reversals, and termination?
7. When an EA is suspended or leaves, do commissions on future transactions by their referred users still accrue? Recommendation: **yes** — attribution is permanent and commissions continue unless EA is terminated for fraud. This makes the referral feel like a real asset the EA is building.

---

## 14. Founder Recommendation

This should be built as a **ledger-backed referral system**, not just a “copy link and split money” feature.

The critical product decision is to separate:
- **attribution**
- **commission calculation**
- **commission payout**

If those are kept as distinct layers, the system will be much easier to debug, reverse, audit, and extend later.

The single best tactical decision for Phase 1 is:
> **Use Stripe Connect Express for onboarding and payouts, but delay transfers until commissions vest instead of transferring instantly at checkout.**

That gives you a fast launch path without painting yourself into an accounting corner.

## 15. Engineering Observations

The following are implementation notes and recommendations from the engineering review of this PRD.

### 15.1 Payout batch table — normalize instead of JSON blob
Section 7.4 stores `stripe_transfer_ids_json` as a JSON column. This will make reconciliation queries ("which commissions were in this batch?", "was this transfer part of a batch?") painful to index and join. Recommend replacing with a join table:

```
ea_payout_batch_items
- id
- batch_id (FK → ea_payout_batches.id)
- commission_id (FK → ea_commissions.id)
- stripe_transfer_id
- amount
- created_at
```

This is not blocking for Phase 1 if batches are small, but should be done before scale.

### 15.2 Hold window recommendation
Open Decision #1 should default to **14 days for card payments, 30 days for ACH/bank**. Stripe's dispute window is 120 days for cards, but the vast majority of chargebacks arrive within 14 days. ACH returns can take longer. Codify this as a configurable per-payment-method setting rather than a single global value.

### 15.3 EA deactivation / offboarding
The PRD covers suspension and fraud reversal, but does not define what happens to the ongoing commission stream when an EA leaves voluntarily or is suspended for non-fraud reasons (e.g., inactivity). Recommended policy:
- **Voluntary departure:** Attribution remains permanent. Commissions on future transactions by referred users continue to accrue and vest. EA can return and collect.
- **Fraud termination:** All `pending` and `vested` commissions are reversed. Future commissions do not accrue. Attribution record is preserved for audit but marked inactive.
- **Inactivity:** After 180 days with no login, EA status moves to `inactive`. Commissions still accrue but payouts are held until EA re-verifies their Stripe account.

### 15.4 Pre-launch readiness — "EA Ready" bug fixes
Jaqulyn is currently evaluating Buy Anything for EA readiness. The following bugs were reported during her evaluation and should be resolved before the EA launch to avoid first-impression churn:

1. **Hero Image Not Loading:** The hero image for "Art Acquisition for New Collectors" isn't loading on the dashboard or via the "Read Guide" link.
2. **Search Results Visibility:** Searching "I need a present for my wife for our silver anniversary" showed a searching message for ~1 minute. Results were hidden until clicking the field. Suggest showing a few results immediately with a "see more" prompt.
3. **Chat Error After Clarifying Question:** After answering a follow-up about personalization/engraving, the system returned a backend error instead of continuing the conversation.
4. **Text Visibility on Guides Page:** On `buy-anything.com/guides`, text colors make content hard to read. Header text only appears on hover, body text can't be highlighted.
5. **Guide Content and Formatting:** Guides lack actionable information. Formatting elements and images not loading properly.
6. **Vendor Directory Relevance:** Searching "house cleaning in Denver" returned mostly unrelated results (bookstore, art museum, airport). Only one actual cleaning service appeared.
7. **Search Results Context:** Searching for a family vacation query returned many links with no explanation for why they were recommended. None of the "Request a Quote" buttons worked.

These are the exact bugs an EA would encounter on their first demo to a potential client. Fixing them is a prerequisite to the Tod outreach campaign.

### 15.5 Codebase Gap Analysis & Implementation Reality
Reviewing the current codebase reveals several specific gaps that must be bridged for this PRD:

1. **Frontend Attribution is Missing:** The main Buy Anything app does not capture `?ref=` on the homepage. Currently, `app/page.tsx` only handles shared search queries.
2. **Auth Storage is Pop-specific:** While the backend auth path (`auth.py`) is reusable and already accepts `ref_code` to set `referred_by_id`, the frontend auth helper (`auth.ts`) is wired specifically to Pop storage keys, not a generalized Buy Anything referral flow.
3. **Login UX:** The login flow (`login/page.tsx`) currently lacks a manual referral-code entry UI for cross-device drop-off cases.
4. **Payment Architecture Mismatch:** Current Buy Anything checkout flows (`checkout.py`, `deals.py`) use connected-account/immediate-transfer patterns for merchants. The EA system requires a completely different paradigm: "platform collects first, ledger records pending commission, background job transfers later after vesting." This decoupling of checkout from payout is the largest backend architectural shift required.
5. **Missing EA Surfaces:** There is currently no `/ea` dashboard surface, no `is_ea`/`ea_status` fields, no commission ledger, no payout batches, and no admin reversal/retry flows. These will all need to be built from scratch.
